# 纯激光定位技术审查

本文审查当前项目里的“纯激光定位”实现，重点覆盖代码细节、运行链路、参数含义、静止时定位细微变化的原因，以及实车场地四周围挡轻微移动对定位的影响。

审查对象主要是：

- `src/origincar_mapping/src/laser_global_localizer.cpp`
- `src/origincar_mapping/launch/localization.launch.py`
- `src/origincar_mapping/scripts/localization_lifecycle_bringup.py`
- `src/origincar_navigation/config/nav2_params.yaml` 中的 `laser_localizer` 段
- `maps/sim_5m_map.yaml` 和 `maps/sim_5m_map.pgm`
- `src/origincar_sim/urdf/origincar_ackermann.urdf.xacro`
- `src/origincar_sim/src/ackermann_kinematic_plugin.cpp`
- `src/origincar_sim/models/5m_map_arena/model.sdf`

## 一、总体结论

当前项目没有使用 AMCL 粒子滤波定位，而是自己实现了一个基于静态栅格地图、2D 激光端点和可选 IMU yaw 先验的全局扫描匹配节点 `laser_global_localizer`。它每次收到 `/scan` 后，在 `/map` 的自由栅格中搜索最匹配的 `map -> base_footprint` 位姿，再结合当前 `odom -> base_footprint` 计算并发布 Nav2 需要的 `map -> odom`。

这套定位更像“每帧独立全局匹配”，不是连续滤波器。代码里没有粒子历史、运动模型预测、低通滤波、静止锁定、位姿滞回或上一帧邻域约束。因此在车辆不动时，只要激光数据、IMU yaw、TF 时间、评分平台或离散搜索结果有一点变化，输出位姿就可能在原地附近细微摆动。

在当前 5m x 5m 正方形仿真环境中，四周墙体几何高度对称、环境特征少，扫描匹配的得分曲面会比较“平”。加上仿真雷达配置了 `0.005m` 标准差的高斯噪声，定位以 `10Hz` 每帧重新全局搜索，输出出现毫米级到厘米级微变是符合当前算法特性的。

如果放到真实场地，四周围挡轻微移动会有影响。影响大小取决于移动幅度、移动范围和地图中可用特征数量：围挡整体只移动几毫米通常只是得分降低或轻微位姿偏置；局部围挡移动 2cm 到 5cm 会更明显；如果围挡是主要甚至唯一定位特征，并且正方形场地内部没有额外静态特征，那么定位会更容易漂、跳或产生系统性偏差。

## 二、启动链路

### 1. 顶层导航启动

`src/origincar_bringup/launch/navigation.launch.py` 按顺序启动：

1. Gazebo 仿真与机器人模型。
2. 延迟 4 秒启动 `origincar_mapping/launch/localization.launch.py`。
3. 延迟 10 秒启动 Nav2。

这样做是为了先让 `/odom`、`/tf`、`/scan` 稳定，再启动定位和导航。

默认地图参数是 `origincar_navigation/maps/sim_5m_map.yaml`。当前工作区根目录也有 `maps/sim_5m_map.yaml`，内容为：

```yaml
image: sim_5m_map.pgm
mode: trinary
resolution: 0.03
origin: [-2.52, -2.52, 0]
negate: 0
occupied_thresh: 0.65
free_thresh: 0.25
```

PGM 文件头显示地图尺寸为 `168 x 168`。结合 `0.03m` 分辨率，地图覆盖约 `5.04m x 5.04m`，和 5m 场地基本一致。

### 2. 定位 launch

`src/origincar_mapping/launch/localization.launch.py` 启动三个东西：

- `nav2_map_server/map_server`：读取地图 YAML，发布 `/map`。
- `origincar_mapping/laser_global_localizer`：执行纯激光定位。
- `localization_lifecycle_bringup.py`：等待并激活 `map_server`。

`map_server` 是生命周期节点，启动后不会立刻发布地图。脚本会调用：

- `/map_server/get_state`
- `/map_server/change_state`

把 `map_server` 从 `unconfigured` 推到 `inactive`，再推到 `active`。脚本只管理 `map_server`，不管理 `laser_localizer`，因为后者是普通 C++ 节点。

### 3. 仿真传感器来源

`src/origincar_sim/launch/sim_world.launch.py` 通过 bridge 转出：

- `/clock`
- `/scan`
- `/imu/data`

`src/origincar_sim/urdf/origincar_ackermann.urdf.xacro` 中，顶部激光是 `gpu_lidar`：

- 话题：`/scan`
- frame：`laser_link`
- 频率：`10Hz`
- 水平采样：`720`
- 角度范围：`-pi` 到 `+pi`，即 360 度
- 量程：`0.05m` 到 `8.0m`
- 距离分辨率：`0.01m`
- 高斯噪声：均值 `0.0`，标准差 `0.005m`

IMU：

- 话题：`/imu/data`
- frame：`imu_link`
- 频率：`50Hz`

底盘插件 `src/origincar_sim/src/ackermann_kinematic_plugin.cpp` 发布：

- `/odom`
- TF `odom -> base_footprint`

插件采用简单阿克曼运动学积分，内部状态是 `x_`、`y_`、`yaw_`。每个仿真步根据命令速度和转向角更新模型位姿，并发布里程计和 TF。车辆不动且没有新速度命令时，速度会在命令超时后变成 0；但定位节点仍会继续根据每帧激光做全局匹配。

## 三、定位节点结构

`laser_global_localizer.cpp` 定义在命名空间 `origincar_mapping` 下。

### 1. 小工具函数

文件开头有三个匿名命名空间函数：

- `DegToRad(double degrees)`：角度转弧度。
- `NormalizeAngle(double angle)`：把角度归一化到 `[-pi, pi]`。
- `RoundStepCells(double step_m, double resolution)`：把米单位搜索步长转换成栅格步长，至少为 1 格。

### 2. 数据结构

`ScanPoint`：

- `x`
- `y`

表示已经转换到 `base_footprint` 下的激光端点。

`Pose2D`：

- `x`
- `y`
- `yaw`

表示二维位姿。

`Candidate`：

- `pose`
- `score`

表示一个候选位姿和它的匹配得分。初始得分是负无穷。

### 3. ROS 节点

类名为 `LaserGlobalLocalizer`，节点名是 `laser_localizer`。

构造函数里初始化：

- `tf2_ros::Buffer`
- `tf2_ros::TransformListener`
- `tf2_ros::TransformBroadcaster`

主函数使用 `rclcpp::executors::MultiThreadedExecutor`，线程数为 2。代码把 scan、TF、map、IMU 放到不同 callback group：

- scan：`MutuallyExclusive`
- TF timer：`Reentrant`
- map：`MutuallyExclusive`
- IMU：`MutuallyExclusive`

这样定位扫描、TF 重发、地图接收和 IMU 接收可以在多线程 executor 中更灵活地调度。

## 五、地图处理细节

### 1. 接收地图

`OnMap()` 接收 `nav_msgs::msg::OccupancyGrid`。

如果地图宽、高为 0，或分辨率小于等于 0，则忽略。否则保存：

- `map_`
- `width_`
- `height_`
- `resolution_`
- `origin_x_`
- `origin_y_`
- `origin_yaw_`

然后调用：

- `BuildDistanceField()`
- `BuildFreeCells()`

最后设置 `map_ready_ = true`。

### 2. 障碍和未知区域定义

`IsObstacleOrUnknown(int8_t value)`：

```cpp
return value < 0 || value >= occupied_threshold_;
```

未知值 `-1` 和占据概率大于等于 `50` 的格子都作为障碍源点。

`IsFree(int8_t value)`：

```cpp
return value >= 0 && value <= free_threshold_;
```

只有值在 `[0, 20]` 的格子才作为可搜索的自由位置。中间概率栅格既不是障碍源点，也不是自由搜索位姿。

### 3. 距离场

`BuildDistanceField()` 会生成两个数组：

- `distance_cells_`
- `distance_scores_`

算法过程：

1. 数组大小是 `width_ * height_`。
2. 所有障碍或未知格子的距离设为 `0`，加入优先队列。
3. 使用 8 邻域 Dijkstra/多源最短路传播距离。
4. 上下左右代价是 `1.0`，对角代价是 `sqrt(2.0)`。
5. 每个格子的距离单位先是“格”，再乘以地图分辨率变成米。
6. 使用高斯函数预计算得分：

```text
score = exp(-(distance_m^2) / (2 * sigma^2))
```

其中：

```text
sigma = max(score_sigma_, resolution_)
```

当前 `score_sigma=0.08m`，地图分辨率 `0.03m`，所以 sigma 为 `0.08m`。

含义是：激光端点越靠近地图障碍或未知边界，得分越接近 1；离障碍越远，得分越接近 0。代码没有显式计算激光束穿越自由空间的概率，只评价“端点是否落在地图障碍附近”。

### 4. 自由搜索格子

`BuildFreeCells()` 遍历整张地图，把 `IsFree()` 的格子保存到 `free_cells_`。粗搜索只会在这些自由格子里枚举机器人位置。

当前地图约 `168 x 168`，总栅格数约 `28224`。已有简版说明中记录自由搜索格子约 `27684`，说明大多数内部区域都是自由空间。

## 六、激光处理细节

### 1. 定位触发

`OnScan()` 是激光回调。

流程：

1. 如果地图还没准备好，节流打印等待 `/map`。
2. 调用 `ShouldRunLocalization()` 控制定位频率。
3. 调用 `BuildScanPoints()` 构造激光端点。
4. 有效点少于 `min_scan_points` 时跳过。
5. 调用 `SearchCoarse()` 粗搜索。
6. 调用 `SearchFine()` 细搜索。
7. 可选发布 debug score。
8. 如果得分低于 `min_score`，保留旧 TF。
9. 调用 `UpdatePoseAndTf()` 更新位姿和 TF。

由于 `/scan` 是 10Hz，`localization_update_frequency` 也是 10Hz，理论上每帧激光都会触发一次定位。

### 2. 频率控制

`ShouldRunLocalization()` 使用 `now()`，不是 scan 消息的 header stamp。它记录 `last_localization_time_`，如果距离上次定位小于周期就返回 false。

周期为：

```text
1 / max(localization_update_frequency_, 0.1)
```

当前是 `0.1s`。

### 3. 雷达坐标变换

`BuildScanPoints()` 先查：

```text
base_footprint <- laser_link
```

如果 scan 的 `header.frame_id` 为空，则用参数 `laser_frame_id`，当前是 `laser_link`。

代码使用：

```cpp
tf_buffer_.lookupTransform(base_frame_id_, source_frame, tf2::TimePointZero)
```

`TimePointZero` 表示取最新可用 TF，而不是取 scan 时间戳对应的 TF。因为 `base_footprint -> laser_link` 是固定关节，这里通常没有问题。

每个激光点计算过程：

1. 按 `scan_step` 每隔若干条射线取一个点。
2. 如果 range 是正无穷，则用 `usable_max_range` 替代。
3. 如果 range 是 NaN、小于 `range_min`、大于 `usable_max_range`，跳过。
4. 在 laser frame 下计算：

```text
lx = range * cos(angle)
ly = range * sin(angle)
```

5. 再通过 `base_from_laser` 的平移和 yaw 转到 `base_footprint`。

当前 720 条射线、`scan_step=10`，理想情况下每帧使用约 72 个端点。

### 4. 无穷远 range 的处理

代码把正无穷 range 当成 `usable_max_range`。这表示没有击中障碍的方向也会生成一个“最大距离端点”。在当前 5m 正方形房间内，大多数方向会打到墙；但在真实场地如果存在开口、透明/低反射材料或量程外区域，这些最大距离端点可能影响评分。

## 七、搜索和评分细节

### 1. 粗搜索

`SearchCoarse()`：

1. 使用 `RoundStepCells(coarse_xy_step_, resolution_)` 把粗搜索步长换成栅格步长。
2. 当前 `0.18 / 0.03 = 6`，所以粗搜索每 6 格取一个自由格子。
3. 对每个符合 stride 的自由格子，调用 `CellCenterToWorld()` 得到世界坐标。
4. 对每个 yaw 候选调用 `ScorePose()`。
5. 保存最高分候选。

代码中 stride 判断是：

```cpp
if ((cell.first % xy_stride) != 0 || (cell.second % xy_stride) != 0) {
  continue;
}
```

这意味着粗搜索采样点和地图 cell index 对齐，而不是和机器人上一帧位置对齐。

### 2. yaw 候选

`BuildCoarseYawSamples()`：

如果 IMU yaw 新鲜：

- 以 IMU yaw 为中心。
- 在 `±imu_yaw_search_window` 内搜索。
- 当前窗口 `30deg`，粗步长 `5deg`。
- 半径步数 `ceil(30 / 5) = 6`。
- 总候选约 `13` 个。

如果没有 IMU、IMU 超时或关闭先验：

- 搜索完整 `[-pi, pi]`。
- 当前粗步长 `5deg`。
- 候选约 `72` 个。

`GetFreshImuYaw()` 判断：

- `use_imu_yaw_prior_` 必须为 true。
- 必须收到过 IMU。
- 如果 `last_imu_time_` 非零，并且 `now() - last_imu_time_ > imu_timeout`，则视为超时。

`OnImu()` 只取 IMU quaternion 的 yaw：

```cpp
latest_imu_yaw_ = NormalizeAngle(tf2::getYaw(msg->orientation));
```

代码没有做 IMU 坐标系到 base 坐标系的额外旋转补偿。当前 URDF 里 `imu_link` 和 `base_link` 是固定同向，仿真中问题不大。实车如果 IMU 安装方向和车体不同，需要保证驱动输出已经对齐，或者修改代码补偿安装外参。

### 3. 细搜索

`SearchFine()` 以粗搜索中心为起点：

- 初始 best 是中心点本身。
- 位置半径：

```text
ceil(coarse_xy_step / fine_xy_step)
```

当前是 `ceil(0.18 / 0.06) = 3`，也就是 x、y 各搜索 `-3` 到 `+3`，一共 7 x 7 个位置。

- yaw 半径：

```text
ceil(coarse_yaw_step / fine_yaw_step)
```

当前是 `ceil(5 / 1) = 5`，也就是 11 个角度。

细搜索只接受位于地图范围内且 `IsFree()` 的机器人位置。端点是否落到自由格子不作为硬限制，只通过距离场得分体现。

### 4. 评分函数

`ScorePose(scan_points, x, y, yaw)`：

1. 把每个 base frame 下的激光点按候选位姿变换到 map/world：

```text
wx = x + cos(yaw) * px - sin(yaw) * py
wy = y + sin(yaw) * px + cos(yaw) * py
```

2. 调用 `WorldToCell()` 转成地图栅格。
3. 如果端点超出地图，跳过加分。
4. 如果在地图内，取 `distance_scores_[index]` 加到总分。
5. 最后除以总激光点数量，而不是除以有效落图点数量。

所以每个候选位姿的得分范围大致是 `[0, 1]`。如果很多端点超出地图，它们相当于贡献 0 分。

### 5. 坐标变换函数

`WorldToCell()` 支持地图 origin 带 yaw 的情况：

```text
dx = wx - origin_x
dy = wy - origin_y
mx = (cos(origin_yaw) * dx + sin(origin_yaw) * dy) / resolution
my = (-sin(origin_yaw) * dx + cos(origin_yaw) * dy) / resolution
cell_x = floor(mx)
cell_y = floor(my)
```

`CellCenterToWorld()` 做反变换，并使用 cell 中心：

```text
mx = (cell_x + 0.5) * resolution
my = (cell_y + 0.5) * resolution
world = origin + R(origin_yaw) * [mx, my]
```

当前地图 origin yaw 为 0，因此就是普通平移和除以分辨率。

## 八、TF 和位姿发布细节

### 1. 输出 `map -> odom`

`UpdatePoseAndTf()` 接收最佳 `map_pose`，即 `map -> base_footprint`。

它再查当前：

```text
odom -> base_footprint
```

使用：

```cpp
tf_buffer_.lookupTransform(odom_frame_id_, base_frame_id_, tf2::TimePointZero)
```

然后计算：

```text
map_from_odom = map_from_base * inverse(odom_from_base)
```

这正是 Nav2 常用 TF 链：

```text
map -> odom -> base_footprint
```

### 2. TF 重发

`PublishStoredTransform()` 独立于定位回调，由 timer 以 `tf_publish_frequency` 重发最近一次 `map -> odom`。当前为 `20Hz`。

发布时间戳是：

```cpp
now() + rclcpp::Duration::from_seconds(transform_tolerance_)
```

当前 `transform_tolerance=0.2s`。这相当于把 TF 稍微发布到未来，减少 Nav2 查询 TF 时因为时间略超前而失败的概率。

### 3. `/laser_pose`

`UpdatePoseAndTf()` 还发布 `geometry_msgs::msg::PoseWithCovarianceStamped`：

- topic：`/laser_pose`
- frame：`map`
- stamp：使用 scan 消息的 header stamp
- position：匹配得到的 `x, y`
- orientation：匹配得到的 yaw

协方差按分数估计：

```text
confidence = clamp(score, 0, 1)
xy_variance = max(0.0025, (1 - confidence) * 0.25)
yaw_variance = max(0.01, (1 - confidence) * 0.50)
```

只填三个对角项：

- `covariance[0]`：x 方差
- `covariance[7]`：y 方差
- `covariance[35]`：yaw 方差

这个协方差没有被当前定位节点内部使用，只作为外部可视化或后续融合的参考。

## 九、当前算法的优点

1. 实现简单直接，依赖少，不需要 AMCL 粒子滤波配置。
2. 使用距离场查表，评分速度快。
3. 支持任意初始位置的全局搜索，不依赖初始位姿。
4. 使用 IMU yaw 先验后，能显著减少正方形环境里的 90 度、180 度方向歧义。
5. 独立 20Hz 重发 TF，Nav2 查询 `map -> odom` 更稳定。
6. 对小地图、低速车、仿真场景足够轻量。

## 十、当前算法的限制和风险

### 1. 每帧独立全局搜索，没有时间连续性约束

定位输出完全由当前 scan 的最高分决定。代码没有使用上一帧位姿作为预测，也没有限制静止时最大跳变量。因此两个相邻 scan 的噪声稍有不同，最高分候选可能在相邻离散位置之间切换。

### 2. 细搜索步长为 0.06m，输出天然有离散跳变

地图分辨率是 `0.03m`，细搜索步长是 `0.06m`。即使真实位姿不动，搜索候选位置也是离散点。若最高分在两个邻近候选之间交替，输出会表现为几厘米级抖动。

### 3. yaw 输出也离散到 1 度附近

细搜索 yaw 步长是 `1deg`。如果场地几何对角度不敏感，或者 IMU yaw 有微小变化，结果可能在相邻 yaw 候选之间切换。

### 4. 只评价激光端点，不评价整条光束

评分只看端点是否靠近地图障碍。真实激光模型通常还会考虑光束经过的自由空间是否和地图一致。当前实现没有这个约束，因此在特征少、对称或局部相似的环境里，会更容易出现多个近似同分位姿。

### 5. 正方形场地特征单一

当前环境是 5m x 5m 正方形，四周都是墙，内部没有明显定位特征。正方形天然存在 90 度和 180 度旋转对称。IMU yaw 可以降低方向歧义，但不能完全消除位置评分平台问题。

### 6. `min_score=0.0` 没有质量门槛

低质量匹配也会更新 TF。仿真里一般还能工作，但实车中如果某次 scan 受动态物体、反光、丢点或围挡变化影响，定位可能接受一个不可靠结果。

### 7. IMU yaw 被直接当作 map yaw 先验

代码默认 IMU yaw 的世界参考和 map yaw 对齐。仿真中通常成立；实车中如果 IMU yaw 是磁北、启动相对角、或者驱动坐标未对齐，yaw 搜索窗口可能把正确方向排除在外。

### 8. TF 查询使用最新时间，不使用 scan 时间

`base_from_laser` 和 `odom_from_base` 都使用 `TimePointZero`。静态外参问题不大，但 `odom_from_base` 是动态 TF。如果机器人运动较快，使用最新 odom 而不是 scan 时间对应 odom，可能引入时序误差。车不动时这个因素较小。

## 十一、为什么车辆不动时定位仍会在原地细微变化

结合当前代码和仿真环境，主要原因如下。

### 1. 仿真激光本身有噪声

URDF 里激光噪声标准差为 `0.005m`，即 5mm。定位每帧只取约 72 个端点参与评分。每个端点都会影响距离场得分，尤其端点靠近墙边时，毫米级距离变化会转化为得分变化。

由于定位每帧都重新搜索最高分，5mm 的 range 噪声可能导致两个相邻候选位姿的分数排名互换。

### 2. 5m 正方形环境得分曲面比较平

四周墙体均匀且对称，缺少角点以外的独特特征。在场地中央或接近对称位置，多个相邻位姿看到的墙面距离变化很小，得分差距也很小。此时少量传感器噪声就能让 best candidate 来回切换。

### 3. 搜索是离散的，不是连续优化

粗搜索每 0.18m 一格，细搜索每 0.06m 一格，yaw 每 1 度一格。输出不做连续插值或曲面拟合。只要最高分格子从一个候选切到另一个候选，`/laser_pose` 就会跳到那个离散候选。

### 4. 没有静止检测和位姿滤波

代码没有读取速度或里程计状态来判断“车现在静止”。即使 `/odom` 速度为 0，定位仍以 10Hz 接受新的 scan 匹配结果。也没有类似：

- 小于阈值的位姿变化不更新
- 静止时降低更新频率
- 对 `/laser_pose` 做低通滤波
- 只在得分明显提升时切换候选
- 使用上一帧位姿做局部跟踪

所以静止时也会“认真地”相信每一帧略有差异的 scan。

### 5. IMU yaw 可能也有微小变化

定位启用了 IMU yaw 先验。IMU 以 50Hz 发布，定位取最新 yaw。仿真 IMU 未显式配置噪声，但仿真数值、姿态解算和 bridge 时间都可能有微小变化。yaw 先验窗口中心一变，粗搜索 yaw 候选集合也随之变化，最终结果可能轻微变动。

### 6. `odom -> base_footprint` 和 `map -> odom` 的组合会放大可见抖动

定位节点输出的是 `map -> odom`，Nav2 和 RViz 常看的机器人位姿是：

```text
map -> odom -> base_footprint
```

如果定位每帧更新 `map -> odom`，即使 `odom -> base_footprint` 本身不动，最终 `map -> base_footprint` 也会跟着定位结果微变。

### 7. 仿真和 TF 时间存在不同发布频率

`/scan` 是 10Hz，`/imu/data` 是 50Hz，定位匹配是 10Hz，TF 重发是 20Hz，底盘插件按仿真步发布 odom/TF。不同数据源的时间戳和最新值组合在一起，静止时一般不会造成大误差，但会带来可观测的微小不一致。

## 十二、真实场地围挡轻微移动的影响

### 1. 为什么围挡会影响当前定位

当前算法的核心假设是：静态地图中的障碍边界和实际激光端点一致。四周围挡就是地图里最主要的障碍边界，也是激光端点得分的主要来源。

如果真实围挡移动了，激光端点会落在“新围挡位置”，而距离场得分仍按“旧地图围挡位置”计算。这样会产生三类影响：

1. 匹配得分降低。
2. 最优位姿发生偏移，让激光端点重新靠近旧地图墙面。
3. 局部移动不一致时，算法无法同时满足所有墙面，可能出现抖动或错误折中。

### 2. 移动幅度和 `score_sigma` 的关系

当前 `score_sigma=0.08m`。距离场得分大致按高斯衰减：

```text
distance = 0cm  -> score = 1.00
distance = 2cm  -> score ~= 0.97
distance = 5cm  -> score ~= 0.82
distance = 8cm  -> score ~= 0.61
distance = 10cm -> score ~= 0.46
```

所以：

- 围挡移动几毫米：一般影响很小，更多表现为得分轻微变化。
- 围挡移动 1cm 到 2cm：通常还能匹配，但会带来小偏差。
- 围挡移动 3cm 到 5cm：已经可能明显影响定位，尤其是在特征少的正方形场地。
- 围挡移动 8cm 以上：和当前 sigma 同量级，匹配质量会明显下降，可能导致偏移或跳变。

### 3. 整体移动和局部移动不同

如果四周围挡作为一个整体刚体平移或旋转，而地图没有同步更新，定位会倾向于把车辆位姿整体反向偏移，使 scan 重新贴合旧地图。这会造成系统性定位偏差。

如果只是某一侧围挡局部弯曲或移动，其他墙不动，算法会在多面墙之间折中。因为评分是所有激光端点平均值，局部变化越大、占用激光点越多，对定位影响越大。

### 4. 正式场地的特殊风险

如果正式场地仍然是四周围挡加空旷内部，定位强依赖围挡。围挡轻微移动、倾斜、布料晃动、临时人员遮挡、反光、低高度导致激光扫不到等情况，都可能带来定位抖动或偏差。

如果正式场地内部有稳定、独特的静态特征，例如固定柱、固定角点、固定障碍物、非对称边界，定位会更稳。因为即使某一段围挡变化，其他特征仍能约束位姿。

## 十三、针对当前项目的改进建议

### 1. 增加静止抑制

可以读取 `/odom` 或 TF 推算速度。当车辆线速度和角速度都接近 0，并且新匹配位姿变化小于阈值时，不更新 `map -> odom`。

建议阈值：

- 平移小于 `0.01m` 到 `0.02m`
- yaw 小于 `0.5deg` 到 `1.0deg`

这样能显著减少原地视觉抖动。

### 2. 增加位姿低通或滞回

当新候选分数只比旧候选略高时，不立即切换。可以设置：

```text
只有 new_score > old_score + score_margin 时才切换到新位姿
```

或者对位姿做低通滤波：

```text
pose = alpha * new_pose + (1 - alpha) * old_pose
```

注意 yaw 要用角度归一化或四元数插值，不能直接跨 `pi` 相加。

### 3. 使用上一帧位姿做局部跟踪

当前每帧都全局搜索。可以改成：

- 初始或丢失时全局搜索。
- 正常跟踪时只在上一帧附近小窗口搜索。
- 得分低或跳变异常时再全局重定位。

这样既降低计算量，也能提高时间连续性。

### 4. 实车提高 `min_score`

当前 `min_score=0.0`，实车不建议长期使用。建议先打开 `debug_publish=true`，记录正常定位分数，再选择一个略低于正常下界的门槛。

例如如果稳定环境下 score 常在 `0.75` 到 `0.90`，可以先尝试 `min_score=0.55` 或 `0.60`，具体以实测为准。

### 5. 调整搜索精度

如果希望定位更细，可以把：

- `fine_xy_step` 从 `0.06` 调到 `0.03`
- `fine_yaw_step_deg` 从 `1.0` 调到 `0.5`

但这会增加计算量，也可能让噪声导致更细碎的抖动。因此最好配合滤波或静止抑制，而不是单独提高精度。

### 6. 增加场地非对称特征

对于 5m x 5m 正方形空场，最有效的工程改进之一是加稳定非对称特征：

- 固定角反射板
- 固定短墙或柱
- 不对称布置的静态标志物
- 稳固、不易移动的定位参考物

这些特征需要同步进入地图，并且比赛/正式运行时保持位置稳定。

### 7. 实车确认 IMU yaw 参考

如果 IMU yaw 不是 map 坐标系下的 yaw，启用 `use_imu_yaw_prior=true` 可能反而把正确方向排除掉。实车前要验证：

```bash
ros2 topic echo /imu/data
ros2 topic echo /laser_pose
```

观察车体实际朝向和定位 yaw 是否一致。必要时：

- 关闭 `use_imu_yaw_prior`
- 增大 `imu_yaw_search_window_deg`
- 在代码中加入 IMU 安装角和初始 yaw offset

## 十四、建议的验证方法

### 1. 查看定位抖动幅度

车辆静止时记录 `/laser_pose`：

```bash
ros2 topic echo /laser_pose
```

重点观察：

- `pose.pose.position.x`
- `pose.pose.position.y`
- orientation 转 yaw 后的角度

建议记录 30 秒，统计最大值、最小值和标准差。

### 2. 打开定位分数

把参数改为：

```yaml
debug_publish: true
```

然后查看：

```bash
ros2 topic echo /laser_localizer/score
```

如果车辆不动但 score 上下波动，同时 pose 微变，基本可以确认是 scan 噪声和离散搜索导致。

### 3. 临时关闭激光噪声验证

在 URDF 中把激光噪声：

```xml
<stddev>0.005</stddev>
```

临时改为：

```xml
<stddev>0.0</stddev>
```

重新启动仿真。如果静止抖动明显减小，说明主要来源是传感器噪声加每帧独立匹配。

### 4. 降低定位频率验证

把：

```yaml
localization_update_frequency: 10.0
```

临时改为：

```yaml
localization_update_frequency: 2.0
```

如果抖动频率变低但幅度类似，说明每次匹配结果本身在跳。

### 5. 真实围挡移动敏感性测试

实车上可以做三组测试：

1. 围挡完全固定，车静止记录 `/laser_pose` 和 score。
2. 单侧围挡平移 1cm 到 2cm，记录变化。
3. 单侧围挡平移 5cm，记录变化。

如果 score 明显下降或 pose 出现系统偏移，就说明当前地图对围挡位置比较敏感，需要更新地图、增大鲁棒性或增加稳定特征。

## 十五、实车落地建议

1. 正式运行前重新建图，确保地图和围挡实际位置一致。
2. 围挡尽量固定，避免比赛过程中被碰撞后不回位。
3. 不要只依赖四周正方形围挡，尽量加入稳定非对称特征。
4. 实车先打开 `debug_publish`，用 score 选择合理 `min_score`。
5. 给定位加静止抑制或低通滤波，减少车停住时 Nav2 看到的 pose 抖动。
6. 确认 IMU yaw 和 map yaw 对齐，否则关闭或放宽 IMU yaw 先验。
7. 若现场存在大量动态障碍，考虑在评分时剔除异常 scan 点，或引入更完整的 AMCL/ICP/NDT/滤波式定位方案。

## 十六、最后判断

当前“车不动但定位在原地细微变化”不是单一 bug，更像是当前算法设计和仿真配置共同导致的正常现象：

- 激光有 5mm 高斯噪声。
- 正方形场地特征少且对称。
- 定位每帧独立全局匹配。
- 搜索结果离散。
- 没有滤波、滞回或静止锁定。

如果只是几毫米到一两厘米的小幅变化，通常可以接受；如果已经影响控制器、局部避障或目标到达判断，建议优先加静止抑制和位姿滞回，再根据实测分数调高 `min_score`。正式场地中，四周围挡的细微移动会影响定位，尤其在围挡是主要特征的情况下。围挡越不稳定，越需要稳定地图、非对称静态特征和更鲁棒的定位更新策略。
