# 实车导航系统分层测试验证手册

本文档用于在 Origincar 小车上逐层验证导航系统，从底层传感器开始，逐步验证到顶层导航。

---

## 系统架构速览

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 9: 全链路导航  (navigation.launch.py)                    │
│  ┌──────────┐  ┌──────────────────┐  ┌───────────────────────┐  │
│  │ 全局规划  │→│ 纯追踪 + 片段避障  │→│ cmd_vel → 底盘MCU     │  │
│  │(route)   │  │(pure pursuit)    │  │                       │  │
│  └──────────┘  └──────────────────┘  └───────────────────────┘  │
├─────────────────────────────────────────────────────────────────┤
│  Layer 5-8: 定位 / 规划 / 控制                                  │
│  laser_global_localizer → map→odom_combined                    │
│  global_costmap + local_costmap                                │
├─────────────────────────────────────────────────────────────────┤
│  Layer 4: 建图  (slam_toolbox)                                  │
│  /scan + /odom_combined → /map                                 │
├─────────────────────────────────────────────────────────────────┤
│  Layer 3: 激光雷达  (lslidar_driver)                            │
│  N10 → /scan (frame: laser_link)                               │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: EKF 融合  (robot_localization)                        │
│  /odom + /imu/data_raw → /odom_combined                        │
├─────────────────────────────────────────────────────────────────┤
│  Layer 1: 底层传感器  (origincar_base)                           │
│  STM32 → /odom, /imu/data_raw                                  │
├─────────────────────────────────────────────────────────────────┤
│  Layer 0: 硬件 / TF 静态变换                                     │
│  串口权限、雷达供电、base_footprint→laser_link 等静态TF          │
└─────────────────────────────────────────────────────────────────┘
```

## Layer 1：底层传感器原始数据

### 1.1 里程计原始数据

**启动：** Layer 0 的底盘节点已启动即可。

**验证：**
```bash
ros2 topic echo /odom --once
```

**检查项：**

| 检查点 | 说明 | 验证方法 |
|--------|------|----------|
| `header.frame_id` | 应为 `odom_combined` | 直接查看 |
| `child_frame_id` | 应为 `base_footprint` | 直接查看 |
| `pose.pose.position` | 往前推车 x 应增大 | 推车同时 echo 观察 |
| `twist.twist.linear.x` | 往前推应为正值 | 推车同时 echo 观察 |
| `twist.twist.angular.z` | 左转应为正值 | 转向同时 echo 观察 |

**RViz 验证：**
```bash
rviz2
```

在 RViz 中：
1. Fixed Frame 设为 `odom_combined`
2. 添加 `TF` 显示，观察 `base_footprint` 是否随推车移动
3. 添加 `Odometry` 显示，topic 选 `/odom`

**通过标准：** 推车移动时，RViz 中的 `base_footprint` 能跟踪实际运动，方向正确，无跳变。

**常见问题：**
- 里程计方向反了：检查小车左右轮定义，或修改 `robot_parameters` 中的坐标系
- 数值不更新：串口通信异常，检查 `/dev/ttyACM0` 是否被其他进程占用

### 1.2 IMU 原始数据

**验证：**
```bash
ros2 topic echo /imu/data_raw --once
```

**检查项：**

| 检查点 | 说明 | 验证方法 |
|--------|------|----------|
| `header.frame_id` | 应为 `gyro_link` | 直接查看 |
| `angular_velocity.z` | 左转应为正值 | 手动旋转小车观察 |
| `linear_acceleration.x` | 前推应为正值 | 前推小车观察 |
| `orientation` | 初始可能为 0 | 正常，等 Madgwick 滤波 |

**RViz 验证：**
在 RViz 中：
1. Fixed Frame 设为 `gyro_link`
2. 点击左下角 `Add` → 搜索 `Imu`，添加 IMU 显示面板
3. topic 选 `/imu/data_raw`
4. 观察 RViz 左侧 Imu 面板中的实时数值

**Imu 面板显示内容：**

| 字段 | 含义 | 验证方式 |
|------|------|----------|
| Orientation (Roll/Pitch/Yaw) | 姿态角 | 旋转小车，对应角度实时变化 |
| Angular Velocity (x/y/z) | 三轴角速度 | 绕 z 轴旋转时 z 值变化 |
| Linear Acceleration (x/y/z) | 三轴加速度 | 前推小车时 x 值变化，静止时 z≈9.8 |

另外也可以：
5. 添加 `TF` 显示，观察 `gyro_link` 的朝向箭头是否跟随小车旋转

**通过标准：** 角速度和加速度方向与实际运动一致，无剧烈跳动。静止时加速度 z 轴接近 9.8 m/s²（重力）。

## Layer 2：EKF 融合里程计

### 2.1 启动 EKF 和 IMU 滤波

**完整启动 Layer 0-2：**
```bash
ros2 launch origincar_bringup robot.launch.py
```

这会启动：底盘串口、静态 TF、N10 雷达、IMU 滤波、EKF。

**验证节点：**
```bash
ros2 node list | grep -E "ekf|imu_filter|origincar_base|lslidar"
```

**预期输出：** 应有 `ekf_filter_node`、`imu_filter_madgwick`、`origincar_base_node`、`lslidar_driver_node`。

### 2.2 验证 EKF 融合输出

```bash
ros2 topic echo /odom_combined --once
```

**检查项：** 与 `/odom` 相同，但数值应更平滑。

**TF 链验证：**
```bash
ros2 run tf2_ros tf2_echo odom_combined base_footprint
```

**RViz 验证：**
1. Fixed Frame 设为 `odom_combined`
2. 添加 TF，观察 `odom_combined` → `base_footprint` 的变换
3. 推车走直线约 2 米，观察轨迹是否直线

**通过标准：**
- 推车走直线，RViz 中轨迹基本为直线，无明显侧向漂移
- 转向 90 度，yaw 角度变化大致正确
- 推车 2 米，里程计显示移动约 2 米（误差 < 10% 即可接受）

**RViz 中验证里程计累积误差：**
1. 推车绕一圈回到原点
2. 观察 `base_footprint` 是否回到原点附近
3. 典型误差：2 米圈内偏移 < 0.3 米

### 2.3 验证 IMU 滤波输出

```bash
ros2 topic echo /imu/data --once
```

**检查项：** `orientation` 四元数应非零，yaw 角应随小车旋转变化。

**RViz 验证：**
1. 添加 `TF`，观察 `base_footprint` → `gyro_link` 方向
2. 旋转小车，`gyro_link` 的朝向应同步变化

**通过标准：** IMU yaw 方向正确，旋转 90 度后 yaw 变化约 1.57 rad。

## Layer 3：激光雷达数据

### 3.1 验证雷达节点

**节点已在 Layer 2 的 `robot.launch.py` 中启动。**

```bash
ros2 topic echo /scan --once
```

**检查项：**

| 检查点 | 说明 | 验证方法 |
|--------|------|----------|
| `header.frame_id` | 应为 `laser_link` | 直接查看 |
| `range_min` / `range_max` | 0.2 / 200.0（N10 默认） | 直接查看 |
| `angle_min` / `angle_max` | 通常 -π 到 +π（360°） | 直接查看 |
| `ranges` 数组 | 应有非空数据 | 直接查看 |

**通过标准：** `ranges` 数组中有实际距离数据（非全 inf），角度范围覆盖完整。

### 3.2 RViz 可视化验证

```bash
rviz2
```

在 RViz 中：
1. Fixed Frame 设为 `laser_link`（或 `base_footprint`）
2. 添加 `LaserScan` 显示，topic 选 `/scan`
3. 观察激光点云是否与实际环境匹配

**验证步骤：**
1. 将小车放在房间中央，观察激光点云是否形成房间轮廓
2. 在雷达前方 1 米处放置障碍物，观察激光点云是否出现对应点
3. 移动障碍物，观察激光点云是否实时更新

**通过标准：**
- 激光点云轮廓与实际墙壁/障碍物形状一致
- 前方 1 米障碍物在激光中显示距离约 1 米
- 激光坐标系方向正确（小车前方对应激光前方）

### 3.3 TF 链完整性验证

```bash
ros2 run tf2_ros tf2_echo base_footprint laser_link
```

**预期输出：** 能看到 `base_footprint` 到 `laser_link` 的静态变换（z 约 0.18m）。

**完整 TF 链：**
```bash
ros2 run tf2_tools view_frames
```

**预期输出：** `odom_combined` → `base_footprint` → `base_link` → `laser_link` 链路完整。

```bash

```

**预期输出：** 能看到 `map` → `odom_combined` 的变换。

### 5.4 定位精度验证

**RViz 验证步骤：**
1. Fixed Frame 设为 `map`
2. 添加 `Map`（topic `/map`）
3. 添加 `LaserScan`（topic `/scan`）
4. 添加 `TF`，观察 `map` → `odom_combined` → `base_footprint`

**关键测试：**

**(a) 静态定位测试：**
1. 将小车放在地图中已知位置
2. 观察激光点云是否与地图轮廓对齐
3. 若不对齐，需要给初始位姿：

```bash
ros2 run tf2_ros static_transform_publisher \
  x y z yaw 0 0 map odom_combined
```

**通过标准：** 激光点云与地图墙壁轮廓完全贴合，偏差 < 0.05m。

**(b) 动态定位测试：**
1. 遥控小车缓慢移动（0.15 m/s）
2. 观察 `map` → `odom_combined` TF 是否平滑更新
3. 观察激光点云是否始终贴合地图

**通过标准：**
- 移动时激光点云始终贴合地图，无漂移
- 无突然跳变

**(c) 定位分数监控：**
```bash
ros2 topic echo /laser_localizer/score
```

**通过标准：** 分数 > 0.35（默认 `min_score` 阈值），稳定在 0.5 以上为佳。

### 5.5 定位恢复测试（"绑架机器人"）

1. 将小车抬起到地图中另一个位置放下
2. 观察是否能自动恢复定位

**通过标准：** 放下后 1-2 秒内激光点云重新贴合地图，定位分数恢复。

## Layer 7：局部控制验证

### 7.1 纯追踪路径跟踪

**RViz 调试话题：**
```
/FollowPath/local_plan          — 当前跟踪路径
/FollowPath/target_pose         — 纯追踪前视点
/FollowPath/goal_reached        — 到达状态
```

在 RViz 中添加：
1. `Path`，topic `/FollowPath/local_plan`（绿色）
2. `Path`，topic `/plan`（蓝色，对比全局路径）
3. `Pose`，topic `/FollowPath/target_pose`

**测试步骤：**

**(a) 直线跟踪：**
1. 在车体正前方 2 米设置目标
2. 观察 `local_plan` 是否贴合全局路径
3. 观察小车是否沿直线行驶到目标

**通过标准：** 小车走直线到达目标，最终位置偏差 < 0.08m。

**(b) 曲线跟踪：**
1. 设置需要转弯的目标点
2. 观察 `target_pose`（前视点）是否沿路径平滑移动
3. 观察小车是否平滑转弯

**通过标准：** 转弯平滑，不剧烈摆动，能到达目标。

**(c) 倒车验证：**
1. 设置目标在车体后方
2. 观察 `cmd_vel.linear.x` 是否为负值

```bash
ros2 topic echo /cmd_vel --once
```

**通过标准：** 倒车速度方向正确，小车向后行驶。

**(d) 终点姿态验证：**
1. 在地图上设置带特定朝向的终点
2. 观察到达后小车朝向是否与目标一致

**通过标准：** 终点 yaw 偏差 < 1.05 rad（默认 `yaw_goal_tolerance`）。

### 7.2 速度平滑验证

```bash
ros2 topic echo /cmd_vel
```

**检查项：** `linear.x` 和 `angular.z` 变化应平滑，无突变。

**通过标准：** 速度命令无明显跳变，加减速自然。

## Layer 8：局部避障验证

### 8.1 启用避障参数

确认 `nav2_params.yaml` 中 `FollowPath` 段：
```yaml
avoidance_enabled: true
```

### 8.2 静态障碍物避障

**测试步骤：**
1. 设置一个需要经过某区域的目标点
2. 在路径中间放置障碍物（纸箱、锥桶等）
3. 观察小车是否能绕开障碍物

**RViz 查看：**
```
/FollowPath/astar_path            — Hybrid A* 绕障路径
/FollowPath/optimized_local_path  — 平滑后的绕障片段
local_costmap/costmap             — 局部代价地图
```

在 RViz 中添加：
1. `Map`，topic `local_costmap/costmap`（观察障碍物是否被正确标记）
2. `Path`，topic `/FollowPath/astar_path`（观察绕障搜索路径）
3. `Path`，topic `/FollowPath/optimized_local_path`（观察平滑后的绕障片段）
4. `Path`，topic `/FollowPath/local_plan`（观察最终跟踪路径）

**通过标准：**
- 障碍物在 `local_costmap` 中被正确标记（红色/紫色）
- `astar_path` 生成了一条绕开障碍的前进路径
- 小车实际绕开了障碍物，未碰撞
- 绕过障碍后，`local_plan` 恢复贴合全局路径
- 全局路径未因障碍物重新规划（此为核心设计特性）

### 8.3 动态障碍物避障

**测试步骤：**
1. 小车沿路径行驶
2. 在路径前方突然放置障碍物
3. 观察小车反应

**通过标准：** 小车在 1-2 秒内检测到障碍并绕过，不会撞上。

### 8.4 避障失败恢复

**测试步骤：**
1. 用紧密排列的障碍物形成无法通过的区域
2. 观察恢复行为

**通过标准：** 行为树触发恢复动作（ClearCostmap → Wait → BackUp），不会无限卡住。

### 8.5 避障参数调优（如需要）

若避障效果不理想，调整 `nav2_params.yaml` 中 `FollowPath` 段：

| 现象 | 调整参数 | 方向 |
|------|----------|------|
| 发现障碍太晚 | `segment_scan_distance` | 增大（如 2.0 → 3.0） |
| 绕障太贴近障碍 | `astar_lethal_cost` | 降低（如 253 → 200） |
| 绕障路径转弯太急 | `hybrid_astar_step_distance` | 减小（如 0.12 → 0.08） |
| 绕障路径绕大圈 | `hybrid_astar_max_path_length_ratio` | 减小（如 3.0 → 2.0） |
| 计算太慢 | `astar_grid_downsample_factor` | 调到 2 |
| 替换片段接不顺 | `segment_pre_margin` | 增大（如 0.30 → 0.50） |
